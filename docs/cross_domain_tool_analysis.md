# Cross-Domain Tool Analysis

Source of truth: `eval_runs/tool_catalog_all.json`, generated by
`eval/export_tool_catalog.py`. The exporter walks
`tau2.registry.get_domains()`, constructs each `Environment`
(`retrieval_variant="bm25"` for `banking_knowledge`), and reads both
`env.tools` and `env.user_tools` via the `ToolKitBase` API (`get_tools()`
+ `get_discoverable_tools()`). No AST parsing.

## 1. Kimi's claim: "only banking_knowledge uses discoverable tools"

**Mostly correct, with one nuance.** Discoverable tools only appear in
`banking_knowledge` — but on *both* of its toolkits, not only the
assistant side:

| Domain              | Assistant discoverable | User discoverable |
|---------------------|-----------------------:|------------------:|
| mock                | 0                      | 0                 |
| airline             | 0                      | 0                 |
| retail              | 0                      | 0                 |
| telecom             | 0                      | 0                 |
| telecom-workflow    | 0                      | 0                 |
| banking_knowledge   | **44**                 | **4**             |

Every non-banking domain decorates tools with plain `@is_tool`, so the
agent and user simulator each see their full toolset from turn 0. The 44
assistant-side discoverables match the prior AST-parser's "approximately
44-48". The 4 user-side discoverables (`deposit_check_3847`,
`get_card_last_4_digits`, `get_referral_link`,
`submit_cash_back_dispute_0589`) are gated by
`list_discoverable_user_tools` / `call_discoverable_user_tool` — a
banking-specific mechanism the AST parser and Kimi both missed.

## 2. Per-domain tool counts

| Domain            | Assistant total | User total | Discoverable (A/U) |
|-------------------|----------------:|-----------:|-------------------:|
| mock              | 4               | 2          | 0 / 0              |
| airline           | 14              | 0          | 0 / 0              |
| retail            | 16              | 0          | 0 / 0              |
| telecom           | 13              | 30         | 0 / 0              |
| telecom-workflow  | 13              | 30         | 0 / 0              |
| banking_knowledge | 59 (15 + 44)    | 10 (6 + 4) | 44 / 4             |

For `banking_knowledge` the split is 15 always-visible + 44 discoverable
on the assistant side, and 6 always-visible + 4 discoverable on the user
side. Other domains have a flat, fully-visible toolset.

## 3. Structural observations

The non-banking domains are dramatically simpler. `airline` and `retail`
expose no user toolkit at all — the user simulator only talks, the
assistant owns every side-effect. `telecom` / `telecom-workflow` flip
that: a 13-tool assistant toolkit plus a 30-tool user toolkit of device
diagnostics (`check_apn_settings`, `check_network_mode_preference`,
`restart_device`, …), all fully visible from turn 0.

`banking_knowledge` is the only domain doing four things at once:
(a) assistant-side discoverable tools gated via
`list_discoverable_agent_tools` / `unlock_discoverable_agent_tool` /
`call_discoverable_agent_tool`; (b) a parallel user-side gate for its 4
user discoverables; (c) per-retrieval-variant toolkit composition (we
only captured `bm25`); and (d) name-mangled numeric suffixes (`*_8291`,
`*_4821`) that look deliberately chosen to defeat string matching. In
retrospect, several banking-specific pieces of `compass.py` look
over-engineered for the common case: the numeric-suffix canonicalization,
the discoverable-vs-visible branching in catalog rendering, and the
unlock state-machine in the playbook all solve problems that simply
don't exist in the other five environments.

## 4. Framework value-prop reassessment

If Kimi is right — and the data confirms she essentially is — then
`compass.py`'s catalog-parsing half is banking-only in practice today.
The machinery that walks an AST, tags tools visible vs. discoverable,
canonicalizes suffixes, and maintains unlock state has exactly one live
user. Measured purely by "how many domains benefit on day 1", the
catalog code scores one.

The other two exported surfaces are a different story. The scenario
playbook + prompt-section API and the canonicalization/normalization
helpers are legitimately domain-agnostic: every domain has a policy
string, a tool list, and a need to render them into a prompt section,
and every domain benefits from consistent trace canonicalization. So
the split PR remains valuable — but its framing should shift. It isn't
"here's a domain-agnostic catalog framework". It's "here's a
domain-agnostic prompt/playbook framework, plus a banking-specific
catalog module that sits next to it". The generic half justifies itself
for airline/retail/telecom even though the catalog half will stay
single-tenant until a second domain adopts `@is_discoverable_tool`.
